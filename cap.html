Implementación de un Modo Demo en Django

Para crear un entorno de demostración seguro y profesional en una aplicación Django, se requiere estructurar la solución en dos niveles principales: una capa de presentación (frontend) que ofrezca un panel de control con accesos directos claros y un aviso informativo, y una capa de seguridad (backend) que bloquee en su totalidad cualquier intento de alteración, creación o eliminación de registros en la base de datos sin interrumpir funciones vitales como la autenticación.

1. La Capa de Presentación y Experiencia de Usuario

El primer paso consiste en definir una vista de bienvenida atractiva que actúe como página de aterrizaje en la raíz del proyecto. Para ello se creó un archivo de plantilla en elements/templates/elements/welcome.html. Esta plantilla aprovecha Bootstrap 5 y clases personalizadas para mostrar tarjetas interactivas de cada uno de los módulos activos de la aplicación (Catálogo de Elementos, Gestión de Libros, Tareas, Comentarios, API Rest y el Panel de Administración de Django).

Para integrar la navegación en el proyecto, se asignó una nueva función de vista en elements/views.py y se mapeó a la URL raíz del proyecto dentro de mystore/urls.py antes del resto de los módulos:

from elements.views import welcome

urlpatterns = [
    path('', welcome, name='welcome'),
    # ... otros módulos
]

Adicionalmente, se modificó el archivo de plantilla base (elements/templates/base.html) para incorporar un contenedor global que renderiza los mensajes del framework de mensajería de Django (django.contrib.messages). De este modo, si un usuario intenta realizar una acción prohibida en el modo de demostración, el sistema inyecta un aviso dinámico de advertencia sin romper la navegación.

2. Capa de Seguridad Intermedia: Middleware de Modo Demo

Para impedir solicitudes que alteren el estado del sistema a través del protocolo HTTP, se implementó un Middleware personalizado en mystore/middleware.py. El middleware intercepta el ciclo de solicitud-respuesta y actúa de la siguiente manera:

  • Métodos Seguros: Permite de forma libre las solicitudes de lectura (GET, HEAD, OPTIONS, TRACE). No obstante, realiza una inspección para bloquear solicitudes de tipo GET dirigidas a rutas de borrado directo (por ejemplo, URLs de elementos que procesan la eliminación mediante parámetros de ruta).
  • Métodos de Mutación: Bloquea las peticiones de tipo POST, PUT, PATCH y DELETE.
  • Excepciones de Autenticación: Excluye explícitamente rutas esenciales de inicio y cierre de sesión (como /login, /logout y el panel administrativo) para que las sesiones de demostración sigan funcionando correctamente.

Si la solicitud es interceptada y bloqueada, el middleware responde inteligentemente: si se trata de una solicitud asíncrona (AJAX/API), retorna una respuesta estructurada en formato JSON con código de estado HTTP 403 Forbidden; de lo contrario, añade un mensaje de advertencia y redirige al usuario a la página anterior o al inicio:

from django.shortcuts import redirect
from django.contrib import messages
from django.http import JsonResponse
from django.urls import resolve, Resolver404

class DemoModeMiddleware:
    def __init__(self, get_response):
        self.get_response = get_response

    def __call__(self, request):
        path = request.path_info
        
        if request.method in ('GET', 'HEAD', 'OPTIONS', 'TRACE'):
            try:
                match = resolve(path)
                view_name = match.view_name
            except Resolver404:
                view_name = ""

            if 'delete' in view_name.lower() or 'delete' in path.lower():
                if not path.startswith('/admin/'):
                    return self.handle_blocked_write(request)
            return self.get_response(request)

        allowed_auth_paths = ['/login', '/logout', '/accounts/login/', '/accounts/logout/', '/admin/login/', '/admin/logout/']
        if any(path.startswith(p) for p in allowed_auth_paths) or 'login' in path.lower():
            return self.get_response(request)

        return self.handle_blocked_write(request)

    def handle_blocked_write(self, request):
        message = "¡Modo Demo Activo! Las operaciones de escritura están desactivadas."
        if request.headers.get('x-requested-with') == 'XMLHttpRequest' or request.path_info.startswith('/api/'):
            return JsonResponse({'detail': message}, status=403)
        messages.warning(request, message)
        referrer = request.META.get('HTTP_REFERER')
        return redirect(referrer if referrer else '/')

Para activar este comportamiento, se registró el middleware dentro del listado MIDDLEWARE en mystore/settings.py justo después de la carga del middleware de autenticación y mensajería.

3. Capa de Seguridad Definitiva: Señales de Bloqueo a Nivel de Base de Datos

Como última línea de defensa contra accesos no autorizados a nivel de persistencia de datos (por ejemplo, escrituras directas desde la consola de comandos manage.py, procesos en segundo plano o desde el propio administrador de Django), se implementaron interceptores del ORM utilizando las señales pre_save y pre_delete en elements/signals.py:

from django.db.models.signals import pre_save, pre_delete
from django.dispatch import receiver
from django.core.exceptions import PermissionDenied

@receiver(pre_save)
def block_saves(sender, instance, **kwargs):
    model_name = sender.__name__
    # Permitir almacenar sesiones
    if model_name in ('Session', 'SessionStore'):
        return
    # Permitir actualización de inicio de sesión de usuarios existentes
    if model_name == 'User' and instance.pk is not None:
        return
    raise PermissionDenied("¡Modo Demo Activo! Creación o modificación deshabilitada.")

@receiver(pre_delete)
def block_deletes(sender, instance, **kwargs):
    if sender.__name__ in ('Session', 'SessionStore'):
        return
    raise PermissionDenied("¡Modo Demo Activo! Eliminación de registros deshabilitada.")

Estas señales se importan dentro del método ready() de la configuración de la aplicación en elements/apps.py, asegurando su carga y activación global durante el arranque de Django. Gracias a este enfoque robusto de doble capa, la aplicación se mantiene segura, íntegra y perfectamente explorable en un ambiente de demostración.
